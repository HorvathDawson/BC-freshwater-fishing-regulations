SCOPE OBJECT TYPES:

1. WHOLE_SYSTEM: Applies to entire waterbody
   - Use when no specific sub-location mentioned

   - location_verbatim: null (UNLESS specific location text like "entire lake" or "all waters" appears in the regulation text)

2. DIRECTIONAL: Subset relative to landmark with clear spatial direction FROM that landmark
   - Examples: "upstream of bridge", "north of powerline", "downstream of dam", "north of a line between Bald and Knob Points", "southeast of a line", "northeast of line"
   - Requires landmark_verbatim AND spatial direction FROM the landmark

   - Must specify movement/position relative to the landmark
   - The landmark can be a line/boundary described with "between" (e.g., "line between X and Y")
   - Do NOT use for regional references like "south end" or "north arm" (use NAMED_PART)
   - Do NOT use when text specifies TWO waterbody segment endpoints (use SEGMENT instead)

3. SEGMENT: Specific stretch between two points
   - Examples: "from Bridge A to Bridge B", "upstream of Eve River, to Hwy 19 bridge", "between Strathcona Dam and Campbell Lake"
   - Requires landmark_verbatim AND landmark_end_verbatim
   - direction: MUST BE "BETWEEN" (never UPSTREAM/DOWNSTREAM for SEGMENT)
   - Use SEGMENT when text has TWO landmarks/bounds, regardless of directional words
   - **CRITICAL**: If location text contains "between X and Y", "from X to Y", or any TWO named landmarks/boundaries, this is ALWAYS a SEGMENT, never WHOLE_SYSTEM
   - **DO NOT** confuse waterbody name with scope location - even if the waterbody is named "X's TRIBUTARIES", the scope can still be a SEGMENT if the regulation text specifies boundaries

4. NAMED_PART: Well-known named subsection or regional reference
   - Examples: "West Arm", "Main Body", "MAINSTEM ONLY", "south end", "north arm"
   - Use for geographic regions, named sections, or parts of waterbodies
   - location_verbatim: the named part reference

     * direction: UPSTREAM or DOWNSTREAM (indicates which side)
   - Requires landmark_verbatim and distance specification in location_verbatim
   - **DISTINCTION**: BUFFER is bounded (specific distance), DIRECTIONAL is unbounded (everything in that direction)

6. TRIBUTARIES_ONLY: Applies strictly to tributary network

SCOPE TYPE DECISION TREE (Apply in this order):

1. Does location text contain "between X and Y" OR "from X to Y" describing the WATERBODY segment?
   → YES: Use SEGMENT type (with landmark_verbatim and landmark_end_verbatim)
   → Example: "Campbell River between Strathcona Dam and Campbell Lake" = SEGMENT
   → EXCEPTION: If "between" describes a LINE/BOUNDARY used in a directional phrase, use DIRECTIONAL
   → Example: "north of a line between Bald and Knob Points" = DIRECTIONAL (not SEGMENT)
   → Look for: "line between", "boundary between" - these describe the landmark, not the segment

2. Does location text specify a BOUNDED distance in one or both directions from a landmark?
   → Check for: "within X m of", "X m upstream and downstream of", "downstream X m from", "from X downstream Y m"
   → YES: Use BUFFER type
   → Two-sided (symmetric): "within X m", "X m upstream and downstream" → direction: null
   → One-sided (bounded in one direction): "downstream 100 m from", "from X downstream Y m" → direction: UPSTREAM or DOWNSTREAM
   → DISTINCTION: BUFFER = bounded/finite distance; DIRECTIONAL = unbounded (everything in that direction)

3. Does location text contain directional word + landmark WITHOUT a distance limit (upstream/downstream/north/south/east/west/northeast/northwest/southeast/southwest of X)?
   → YES: Use DIRECTIONAL type (with landmark_verbatim and direction)
   → CRITICAL: If text says "downstream 100 m from X", this is BUFFER (bounded), not DIRECTIONAL
   → DIRECTIONAL means EVERYTHING in that direction (unbounded): "upstream of dam" = all water upstream, "southeast of a line" = all water southeast
   → Ordinal directions: Use NORTHEAST_OF, NORTHWEST_OF, SOUTHEAST_OF, SOUTHWEST_OF for diagonal/ordinal directions
   → NOTE: If it mentions TWO waterbody endpoints (not just a line between points), revert to SEGMENT
   → Example: "upstream of X to Y" = SEGMENT, but "upstream of X" = DIRECTIONAL

4. Does location text reference a named region/part (Main Body, West Arm, north end, south arm)?
   → YES: Use NAMED_PART type
   → **CRITICAL EXCEPTION**: If text contains "within X m/km of [landmark]", this is BUFFER, NOT NAMED_PART
     * "within 200 m of Bush-Sullivan Road Bridge in Bush Arm" = BUFFER (has distance)
     * "at Bush Arm" = NAMED_PART (no distance specification)

5. Does location text say "tributaries only" or exclude mainstem explicitly?
   → YES: Use TRIBUTARIES_ONLY type

6. Is there NO specific location mentioned, or does it say "all", "entire", with no boundaries?
   → YES: Use WHOLE_SYSTEM type
   → NO: If you see ANY location specification at all, DO NOT use WHOLE_SYSTEM
   → **CRITICAL**: If the waterbody NAME ends with "'S TRIBUTARIES" but the RULE TEXT specifies a segment/location, the SCOPE is NOT WHOLE_SYSTEM
     * Example: "(Lower) CAMPBELL LAKE'S TRIBUTARIES" with rule "Campbell River between Strathcona Dam and (Lower) Campbell Lake"
     * This is SEGMENT (the rule specifies boundaries), NOT WHOLE_SYSTEM (even though waterbody name mentions tributaries)
   → **NEVER use WHOLE_SYSTEM** if location_verbatim contains: "between", "from...to", "upstream of", "downstream of", "within X m", or any directional/distance language

7. Is the location text genuinely ambiguous, contradictory, or impossible to classify confidently?
   → YES: Use VAGUE type (LAST RESORT ONLY)
   → Document the ambiguity in a restriction note
   → Example: "near the area" (unclear landmark), "within 100 m" (of what?)
   → **CRITICAL**: If location contains multiple distinct geometries like "upstream and downstream of X", create TWO separate rule groups instead of using VAGUE

**CRITICAL**: The waterbody NAME in the title is INDEPENDENT from the scope LOCATION in the regulation text.
- Even if waterbody is named "X's TRIBUTARIES", the scope can be SEGMENT, DIRECTIONAL, etc.
- Always analyze the regulation TEXT to determine scope type, not just the waterbody name.

IDENTITY PARSING - COMPREHENSIVE STRUCTURE:

**IDENTITY_TYPE CLASSIFICATION**:

Classify the waterbody type based on the name. Valid types:
- STREAM: All flowing water bodies including rivers, creeks, streams, watersheds, and their named parts
  * Examples: "ELK RIVER", "BURNT BRIDGE CREEK", "COQUIHALLA RIVER", "LIARD RIVER WATERSHED", "ATNARKO/BELLA COOLA RIVERS"
  * Note: "WATERSHED" in the name is just STREAM type - the tributary inclusion is captured by global_scope.includes_tributaries
- STILL_WATER: All standing/non-flowing water bodies including lakes, reservoirs, ponds, sloughs, bays, arms
  * Examples: "KOOTENAY LAKE", "KINBASKET LAKE", "NATION ARM", "DAVIS BAY", "LEWIS SLOUGH", "DOWNTON LAKE (Reservoir)"
- MANAGEMENT_AREA: Wildlife management areas (e.g., "CRESTON VALLEY WILDLIFE MANAGEMENT AREA")
- PARK: Park waters (e.g., "KIKOMUN CREEK PARK")
- CONFLUENCE: Specific confluence points where waterbodies meet (e.g., "SKEENA RIVER/KISPIOX RIVER CONFLUENCE")
- TRIBUTARIES: When title explicitly about tributary systems (e.g., "ELK RIVER'S TRIBUTARIES")
- MULTIPLE_WATERBODIES: When title covers multiple distinct waterbodies managed together (e.g., "WEAVER LAKE and WEAVER CREEK")
- WATERS: Generic waters designation when type cannot be determined from name

- Alternate names: "(McNaughton)", "(formerly X)" → goes to alternate_names

- Part names that ARE the core identifier: "MAIN BODY", "UPPER WEST ARM" → part of waterbody_key

**EXAMPLES**:
- "ECHOES LAKE (near Kimberley)" → location_descriptor: "near Kimberley"
- "BIG LAKE (approx. 10 km west of 100 Mile House)" → location_descriptor: "approx. 10 km west of 100 Mile House"
- "LARRY LAKE (unnamed lake located about 400 m west of Thalia Lake)" → location_descriptor: "unnamed lake located about 400 m west of Thalia Lake"
- "BOAR LAKE (Dog Creek drainage)" → location_descriptor: "Dog Creek drainage"
- "SQUARE LAKE (located in Crooked River Provincial Park)" → location_descriptor: "located in Crooked River Provincial Park"
- "ELK RIVER (upstream of Elko Dam)" → location_descriptor: null (scope, not location)
- "NATION ARM (Williston Lake)" → location_descriptor: "Williston Lake"
- No location info → location_descriptor: null

**NOTES EXTRACTION**:

Extract administrative/reference information that doesn't fit other categories.

**WHAT TO EXTRACT**:
- Map references: "(for location see map on page 34)", "(see map on page 42)"
- Permit notices: "(permit required see Note on page 34)"
- Cross-references: "(see also east White & North White Rivers)"
- Regulatory notices: "(Angling regulations... currently under review...)"
- Part descriptions when informational: "(all lakes in the park)"

**EXAMPLES**:
- "DUCK LAKE (permit required see Note on page 34)" → notes: "permit required see Note on page 34"
- "KOOTENAY LAKE - MAIN BODY (for location see map on page 34)" → notes: "for location see map on page 34"
- "WHITE RIVER (see also east White & North White Rivers)" → notes: "see also east White & North White Rivers"
- "KIKOMUN CREEK PARK (all lakes in the park)" → notes: "all lakes in the park"
- No notes → notes: null

**EXCLUSIONS EXTRACTION**:

Extract geographic areas that are EXPLICITLY EXCLUDED from the waterbody definition. This is an ARRAY of ScopeObject items.

**THREE SOURCES FOR EXCLUSIONS**:

1. **FROM TITLE** - Direct exceptions in waterbody name:
   - "ATNARKO/BELLA COOLA RIVERS EXCEPT: Burnt Bridge Creek upstream of Sitkatapa Creek, Hunlen Creek upstream of Hunlen Falls"
   - "FRASER RIVER (except tributaries in parks)"
   - Parse each exception as a separate ScopeObject

2. **FROM REGULATION TEXT - "(see exceptions)" pattern**:
   - **CRITICAL**: Title phrase "(see exceptions)" means look for "EXCEPTIONS - SEE SEPARATE LISTINGS FOR:" in raw_regs text
   - Parse ALL waterbodies listed after this marker as exclusions
   - Each listed waterbody becomes a separate exclusion ScopeObject
   - Extract any scope modifiers (upstream/downstream/between) for each exception

3. **FROM REGULATION TEXT - "Does not include" pattern**:
   - Regulation text may state "Does not include [waterbody]" or similar phrasing
   - Extract the waterbody mentioned after "Does not include"
   - Parse as exclusion ScopeObject with appropriate scope type
   - Example: "Does not include the Kootenay River upstream from Kootenay Lake to the U.S. border near Creston"

**PARSING "SEE EXCEPTIONS" PATTERN**:

When waterbody name contains "(see exceptions)":
1. Search raw_regs for "EXCEPTIONS - SEE SEPARATE LISTINGS FOR:"
2. Extract the comma-separated list of waterbodies that follows
3. For each waterbody in the list:
   - Extract waterbody name
   - Extract any directional/scope modifiers ("upstream of", "downstream of", "between X and Y")
   - Create a ScopeObject with appropriate type (DIRECTIONAL, SEGMENT, WHOLE_SYSTEM, etc.)
4. Handle additional nested exceptions (e.g., "EXCEPT Coal Creek downstream of...")

**TRIBUTARY MARKING IN EXCEPTIONS**:
- Text may mark some exceptions with "**[Includes Tributaries]**" (e.g., "Alexander Cr.**[Includes Tributaries]** upstream of...")
- Set includes_tributaries: true for those specific exceptions
- Others default to includes_tributaries: false

**NESTED EXCEPTIONS**:
- Text may have additional exceptions within the exceptions list (e.g., "EXCEPT Coal Creek downstream of old MF&M Railway Bridge")
- These are NOT exclusions - they exclude from the exclusion (double negative = inclusion)
- Handle carefully: if "ALL tributaries (EXCEPT X)" appears, X is actually INCLUDED in the main scope

**EXAMPLES - Title-based exclusions**:
- "ATNARKO/BELLA COOLA RIVERS EXCEPT: Burnt Bridge Creek upstream of Sitkatapa Creek" →
  exclusions: [{type: "DIRECTIONAL", waterbody_key: "Burnt Bridge Creek", location_verbatim: "Burnt Bridge Creek upstream of Sitkatapa Creek", ...}]
- "FRASER RIVER (except Harrison Lake)" →
  exclusions: [{type: "WHOLE_SYSTEM", waterbody_key: "Harrison Lake", location_verbatim: "Harrison Lake", ...}]
- No exclusions → exclusions: []

**INCLUSIONS EXTRACTION**:

Extract geographic areas that are EXPLICITLY INCLUDED in the waterbody definition. This is an ARRAY of ScopeObject items.

**WHEN TO USE**:
- Title contains "(includes X)", "(including X, Y, Z)"
- Title contains "(Includes the reservoirs behind...)"

Each inclusion becomes a ScopeObject with:
- type: Usually NAMED_PART or WHOLE_SYSTEM depending on the inclusion
- waterbody_key: The name of the included waterbody
- location_verbatim: The exact phrase from name_verbatim describing the inclusion
- Other fields as appropriate

**EXAMPLES**:
- "PEND D'OREILLE RIVER (Includes the reservoirs behind Waneta Dam and Seven Mile Dam)" →
  inclusions: [
    {type: "NAMED_PART", waterbody_key: "PEND D'OREILLE RIVER", location_verbatim: "the reservoirs behind Waneta Dam and Seven Mile Dam", ...}
  ]
- "SUMALLO RIVER (includes "Cedar" Lake, at Sunshine Valley)" →
  inclusions: [
    {type: "WHOLE_SYSTEM", waterbody_key: ""Cedar" Lake", location_verbatim: ""Cedar" Lake, at Sunshine Valley", ...}
  ]
- "VASEUX LAKE (including two lagoons on the west side of Okanagan River upstream of McIntyre Dam)" →
  inclusions: [
    {type: "NAMED_PART", waterbody_key: "VASEUX LAKE", location_verbatim: "two lagoons on the west side of Okanagan River upstream of McIntyre Dam", ...}
  ]
- No inclusions → inclusions: []

**VALIDATION**: Verify waterbody_key is derivable from name_verbatim by removing parenthetical sections and suffixes

TRIBUTARY INCLUSION:

**GLOBAL SCOPE** (identity level):
- Inspect the input `symbols` array for markers like "Incl. Tribs", "Includes Tributaries", or similar
- If present, set global_scope.includes_tributaries to `true`
- Check waterbody names that END with "TRIBUTARIES" or "'S TRIBUTARIES" - these also indicate `includes_tributaries: true`
- **The global scope defines the DEFAULT tributary inclusion for all rule scopes**

**RULE SCOPE** (individual rule level):
- **DEFAULT**: Each rule's scope INHERITS includes_tributaries from the global scope
  * If global_scope.includes_tributaries = true → rule scope defaults to true
  * If global_scope.includes_tributaries = false → rule scope defaults to false
  
- **ONLY OVERRIDE when regulation text has EXPLICIT tributary mentions**:
  * "TRIBUTARIES ONLY" → use TRIBUTARIES_ONLY scope type
  * "on TRIBUTARIES ONLY" → includes_tributaries=true (overrides global false if needed)
  * "MAINSTEM ONLY" → includes_tributaries=false (overrides global true if needed)
  * "on mainstem" / "on [River Name] only" → includes_tributaries=false
  * No explicit mention → USE GLOBAL DEFAULT (do not assume based on restriction type)

- **CRITICAL**: Do NOT assume harvest rules include tributaries or vessel rules exclude them
  * ALL rules inherit from global scope unless text explicitly overrides
  * Two WHOLE_SYSTEM rules with different includes_tributaries are DIFFERENT scopes and must be separate rule groups

**Example Scenarios**:
1. Header has "Incl. Tribs" (global=true), text says "Trout quota = 1" (no location mention)
   → Rule scope: includes_tributaries=true (inherits from global)
   
2. Header has "Incl. Tribs" (global=true), text says "No powered boats on mainstem"
   → Rule scope: includes_tributaries=false (explicit "mainstem" overrides global)
   
3. Header has NO "Incl. Tribs" (global=false), text says "Bait ban on tributaries only"
   → Rule scope: includes_tributaries=true (explicit "tributaries only" overrides global)

RESTRICTION TYPES:

- closure: "No Fishing", fishing bans
- harvest: Catch limits, size limits, catch and release
- gear_restriction: Bait ban, fly fishing only, single barbless hook
- vessel_restriction: No powered boats, vessel speed limits
- licensing: Class II water, permits required
- note: Advisory information, references to other sections

GROUPING RULES WITH SAME SCOPE:

**CRITICAL**: Minimize the number of rule groups by combining restrictions with identical scopes, EVEN IF the text is not contiguous.

**YOU MUST NEVER CREATE MULTIPLE RULES WITH IDENTICAL SCOPES**. This is a validation error.

When multiple restrictions apply to the same geographic area, create ONE rule group with:
- rule_text_verbatim: Array containing ALL text segments for that scope (may be non-contiguous)
- scope: Single scope object (not an array)
- restrictions: Array of all restrictions that apply to that scope

**INTERLEAVED TEXT HANDLING**:
When rules for the same scope are separated by other text in the regulation:
- Do NOT create separate rule groups
- Capture each text block in separate array elements of rule_text_verbatim
- Example: "No fishing upstream. Bait ban downstream. Fly fishing only upstream."
  * ONE rule for Upstream: rule_text_verbatim: ["No fishing upstream", "Fly fishing only upstream"]
  * ONE rule for Downstream: rule_text_verbatim: ["Bait ban downstream"]

**SCOPE EQUALITY TEST** - Two scopes are the SAME if ALL of these match:
1. type (both WHOLE_SYSTEM, both DIRECTIONAL, etc.)
2. location_verbatim (exact same text or both null)
3. landmark_verbatim (exact same text or both null)
4. direction (exact same direction or both null)
5. includes_tributaries (both true or both false)

**VALIDATION WILL FAIL** if you create multiple rules with identical scopes. Always combine them.

**CRITICAL**: A BUFFER scope (bounded distance) is NEVER equal to a DIRECTIONAL scope (unbounded), even if they share the same landmark and direction. Keep them separate.
- Example: "upstream of dam" (DIRECTIONAL, unbounded) ≠ "upstream 100 m from dam" (BUFFER, bounded)

Example: "Trout catch and release; bait ban; fly fishing only" all for "upstream of dam"
- ONE rule group with scope="upstream of dam" and 3 restrictions

Example: Two WHOLE_SYSTEM scopes with same tributary status
- "No fishing Apr 1-May 31" + "Bait ban" + "Single barbless hook"
- ONE rule group with WHOLE_SYSTEM scope and 3 restrictions

**Only create SEPARATE rule groups when:**
1. Scopes have different types (WHOLE_SYSTEM vs DIRECTIONAL)
2. Scopes have different locations ("upstream of dam" vs "downstream of dam")
3. Scopes have different tributary inclusion (one includes tribs, one doesn't)
4. Scopes have different landmarks or directions

**ANTI-PATTERN - DO NOT SPLIT UNNECESSARILY**:
❌ WRONG: Creating 2+ WHOLE_SYSTEM rules for restrictions that apply to the same scope
   Example: Rule 1 WHOLE_SYSTEM (vessel_restriction), Rule 2 WHOLE_SYSTEM (note)
   This creates duplicate scope objects needlessly
✓ RIGHT: ONE WHOLE_SYSTEM rule with BOTH restrictions in the array
   Example: Rule 1 WHOLE_SYSTEM (vessel_restriction, note)

**CONSISTENCY PRINCIPLE**: If you have multiple restrictions with the EXACT same scope (use 5-point equality test), they MUST be in ONE rule group. Do not create separate rules unless scopes differ.

Example of when to separate: "No fishing upstream of dam; bait ban downstream of dam"
- TWO rule groups (different locations/directions)

Example of when to separate: WHOLE_SYSTEM rules where one includes tributaries
- "Trout quota = 1" with global "Incl. Tribs" = WHOLE_SYSTEM with includes_tributaries=true (inherits from global)
- "No angling from powered boats on mainstem" = WHOLE_SYSTEM with includes_tributaries=false (explicit "mainstem" override)
- TWO separate rule groups (different tributary inclusion)

**CRITICAL TRIBUTARY LOGIC**:
The global_scope.includes_tributaries defines the DEFAULT for all rule scopes.

**Rule scope inheritance**:
- **DEFAULT BEHAVIOR**: Rule scopes INHERIT includes_tributaries from global scope
  * If global has "Incl. Tribs" symbol → rule defaults to includes_tributaries=true
  * If global has NO "Incl. Tribs" → rule defaults to includes_tributaries=false
  
- **OVERRIDE ONLY with explicit text in regulation**:
  * "TRIBUTARIES ONLY" → use TRIBUTARIES_ONLY scope type
  * "on tributaries" / "tributaries only" → includes_tributaries=true (override global false)
  * "MAINSTEM ONLY" / "on mainstem" / "on [River Name] only" → includes_tributaries=false (override global true)
  * No explicit mention → **USE GLOBAL DEFAULT** (do not infer from restriction type)

**CRITICAL**: Do NOT make assumptions based on restriction type (harvest vs vessel).
- All rules inherit from global unless text explicitly says otherwise
- "Trout quota = 1" with no location + global "Incl. Tribs" = includes_tributaries=true (inherited)
- "No powered boats" with no location + global "Incl. Tribs" = includes_tributaries=true (inherited, not false!)
- Only change when text explicitly mentions "mainstem", "tributaries only", etc.

Example: Header "Incl. Tribs" (global=true)
- "Trout quota = 1" → WHOLE_SYSTEM with includes_tributaries=true (inherits)
- "No powered boats" → WHOLE_SYSTEM with includes_tributaries=true (inherits, same scope as above - COMBINE!)
- "Bait ban on mainstem only" → WHOLE_SYSTEM with includes_tributaries=false (explicit override - SEPARATE!)

Example: Header NO "Incl. Tribs" (global=false)
- "Trout quota = 1" → WHOLE_SYSTEM with includes_tributaries=false (inherits)
- "Catch and release on tributaries" → scope with includes_tributaries=true (explicit override - SEPARATE!)
- "Bait ban" (no trib mention) = WHOLE_SYSTEM with includes_tributaries=false
- TWO separate rule groups
