     * Cannot extract two distinct sign landmarks as verbatim substrings
     * Use VAGUE and document: "Cannot distinguish between upstream and downstream fishing boundary signs as separate verbatim substrings - both reference Cranberry River Canyon"
   - location_verbatim: Copy the confusing text exactly as it appears
   - landmark_verbatim: null (or the best verbatim landmark you can extract, if any)
   - direction: null (or appropriate direction if clear)
   - **REQUIRED**: If type is VAGUE, you MUST add an entry to the top-level `audit_log` array explaining:
     * WHY the location could not be parsed into a specific type
     * WHAT you believe the landmarks/boundaries are (even if not verbatim extractable)
     * Example: "Rule 3: Cannot extract verbatim landmarks for 'between fishing boundary signs upstream of and downstream of Cranberry River Canyon' - text describes two signs (one upstream, one downstream of canyon) but both reference the same canyon without distinct identifiers"
   - Examples of acceptable VAGUE cases:
     * Unclear landmarks: "near the thing", "around the area"
     * Incomplete phrases: "within 100 m" (of what?)
     * Non-extractable landmarks: "between signs upstream and downstream of X" (can't extract two distinct sign substrings)
     * Complex polygon boundaries: "area bounded by a line commencing at a fishing boundary sign located at the eastern end of Landstrom Bar (Scale Bar) to a fishing boundary sign on the opposite bank of the river, thence to a fishing boundary sign at the southern end of Croft Island, thence to a fishing boundary sign on the nearest bank of the river in a westerly direction, thence following the bank of the river in a westerly direction to the Point of Commencement" (defines a polygon with multiple boundary points)
   - **DO NOT USE** just because parsing is difficult - try other types first
   - **DO NOT USE** for multiple distinct geometries - if location string contains "upstream and downstream of bridge" or similar, create TWO separate rule groups (one UPSTREAM, one DOWNSTREAM) instead of marking as VAGUE

SCOPE TYPE DECISION TREE (Apply in this order):

1. Does location text contain "between X and Y" OR "from X to Y" describing the WATERBODY segment?
   → YES: Use SEGMENT type (with landmark_verbatim and landmark_end_verbatim)
   → Example: "Campbell River between Strathcona Dam and Campbell Lake" = SEGMENT
   → EXCEPTION: If "between" describes a LINE/BOUNDARY used in a directional phrase, use DIRECTIONAL
   → Example: "north of a line between Bald and Knob Points" = DIRECTIONAL (not SEGMENT)
   → Look for: "line between", "boundary between" - these describe the landmark, not the segment

2. Does location text specify a BOUNDED distance in one or both directions from a landmark?
   → Check for: "within X m of", "X m upstream and downstream of", "downstream X m from", "from X downstream Y m"
   → YES: Use BUFFER type
   → Two-sided (symmetric): "within X m", "X m upstream and downstream" → direction: null
   → One-sided (bounded in one direction): "downstream 100 m from", "from X downstream Y m" → direction: UPSTREAM or DOWNSTREAM
   → DISTINCTION: BUFFER = bounded/finite distance; DIRECTIONAL = unbounded (everything in that direction)

3. Does location text contain directional word + landmark WITHOUT a distance limit (upstream/downstream/north/south/east/west/northeast/northwest/southeast/southwest of X)?
   → YES: Use DIRECTIONAL type (with landmark_verbatim and direction)
   → CRITICAL: If text says "downstream 100 m from X", this is BUFFER (bounded), not DIRECTIONAL
   → DIRECTIONAL means EVERYTHING in that direction (unbounded): "upstream of dam" = all water upstream, "southeast of a line" = all water southeast
   → Ordinal directions: Use NORTHEAST_OF, NORTHWEST_OF, SOUTHEAST_OF, SOUTHWEST_OF for diagonal/ordinal directions
   → NOTE: If it mentions TWO waterbody endpoints (not just a line between points), revert to SEGMENT
   → Example: "upstream of X to Y" = SEGMENT, but "upstream of X" = DIRECTIONAL

4. Does location text reference a named region/part (Main Body, West Arm, north end, south arm)?
   → YES: Use NAMED_PART type
   → **CRITICAL EXCEPTION**: If text contains "within X m/km of [landmark]", this is BUFFER, NOT NAMED_PART
     * "within 200 m of Bush-Sullivan Road Bridge in Bush Arm" = BUFFER (has distance)
     * "at Bush Arm" = NAMED_PART (no distance specification)

5. Does location text say "tributaries only" or exclude mainstem explicitly?
   → YES: Use TRIBUTARIES_ONLY type

6. Is there NO specific location mentioned, or does it say "all", "entire", with no boundaries?
   → YES: Use WHOLE_SYSTEM type
   → NO: If you see ANY location specification at all, DO NOT use WHOLE_SYSTEM
   → **CRITICAL**: If the waterbody NAME ends with "'S TRIBUTARIES" but the RULE TEXT specifies a segment/location, the SCOPE is NOT WHOLE_SYSTEM
     * Example: "(Lower) CAMPBELL LAKE'S TRIBUTARIES" with rule "Campbell River between Strathcona Dam and (Lower) Campbell Lake"
     * This is SEGMENT (the rule specifies boundaries), NOT WHOLE_SYSTEM (even though waterbody name mentions tributaries)
   → **NEVER use WHOLE_SYSTEM** if location_verbatim contains: "between", "from...to", "upstream of", "downstream of", "within X m", or any directional/distance language

7. Is the location text genuinely ambiguous, contradictory, or impossible to classify confidently?
   → YES: Use VAGUE type (LAST RESORT ONLY)
   → Document the ambiguity in a restriction note
   → Example: "near the area" (unclear landmark), "within 100 m" (of what?)
   → **CRITICAL**: If location contains multiple distinct geometries like "upstream and downstream of X", create TWO separate rule groups instead of using VAGUE

**CRITICAL**: The waterbody NAME in the title is INDEPENDENT from the scope LOCATION in the regulation text.
- Even if waterbody is named "X's TRIBUTARIES", the scope can be SEGMENT, DIRECTIONAL, etc.
- Always analyze the regulation TEXT to determine scope type, not just the waterbody name.

IDENTITY PARSING - COMPREHENSIVE STRUCTURE:

The identity object has multiple fields that work together to fully capture all information in the waterbody title:

1. **name_verbatim**: Exact character-for-character title (already covered above)
2. **waterbody_key**: Core waterbody name ONLY (no scope, location, notes, or modifiers in parentheses)
3. **identity_type**: Waterbody classification (STREAM, STILL_WATER, ADMINISTRATIVE_AREA, CONFLUENCE, TRIBUTARIES, MULTIPLE_WATERBODIES)
4. **alternate_names**: Array of alternate/former/quoted names found in parentheses
5. **location_descriptor**: Disambiguating location information (e.g., "near Kimberley", "approx. 10 km west of 100 Mile House")
6. **notes**: Administrative references (e.g., "permit required see Note on page 34", "see map on page 42")
7. **global_scope**: Spatial constraint from title (e.g., upstream/downstream, specific segment)
8. **exclusions**: Geographic exclusions (e.g., "(see exceptions)", "(except X)")
9. **inclusions**: Geographic inclusions (e.g., "(includes Little Shuswap Lake, Mara Lake)")

**WATERBODY_KEY EXTRACTION**:

**CRITICAL RULES**:
- waterbody_key is the CORE NAME ONLY - extract by removing parenthetical content but keeping ALL text before AND after parentheses
- **STEP 1**: Remove ALL parentheses and their contents (e.g., "ARROW PARK (Mosquito) CREEK" → "ARROW PARK CREEK")
- **STEP 2**: Then remove "'S TRIBUTARIES", "TRIBUTARIES", " WATERSHED", " WATERS" suffixes if present
- **CRITICAL**: Do NOT stop at the text before parentheses - include the TYPE suffix (CREEK, LAKE, RIVER) that comes AFTER the parentheses
  * WRONG: "ARROW PARK (Mosquito) CREEK" → "ARROW PARK" ❌
  * RIGHT: "ARROW PARK (Mosquito) CREEK" → "ARROW PARK CREEK" ✓
  * WRONG: "BIG FISH (Dunbar) LAKE" → "BIG FISH" ❌
  * RIGHT: "BIG FISH (Dunbar) LAKE" → "BIG FISH LAKE" ✓
- Preserve descriptive modifiers that are part of the core name: "Little", "Big", "Upper", "Lower", "North", "South", "East", "West"
- Keep quoted names if they're part of the core (e.g., ""ANDERSON" LAKE"), but remove parenthetical quoted names
- waterbody_key should be CLEAN - just the essential waterbody identifier without alternate names or location info

**EXAMPLES**:
- "ELK RIVER (upstream of Elko Dam)" → waterbody_key: "ELK RIVER"
- "ECHOES LAKE (near Kimberley)" → waterbody_key: "ECHOES LAKE"
- "KINBASKET (McNaughton) LAKE" → waterbody_key: "KINBASKET LAKE" (note: NOT "KINBASKET (McNaughton) LAKE")
- "LITTLE MITTEN" LAKE (approx. 400 m west of Mitten Lake)" → waterbody_key: ""LITTLE MITTEN" LAKE"
- "LARRY LAKE (unnamed lake located about 400 m west of Thalia Lake)" → waterbody_key: "LARRY LAKE"
- "KSI SII AKS RIVER (formerly Tseax River)" → waterbody_key: "KSI SII AKS RIVER"
- "SKEENA RIVER/KISPIOX RIVER CONFLUENCE" → waterbody_key: "SKEENA RIVER/KISPIOX RIVER CONFLUENCE"
- "CRESTON VALLEY WILDLIFE MANAGEMENT AREA (CVWMA) WATERS" → waterbody_key: "CRESTON VALLEY WILDLIFE MANAGEMENT AREA"
- "KOOTENAY LAKE - MAIN BODY (for location see map on page 34)" → waterbody_key: "KOOTENAY LAKE - MAIN BODY"
- "NATION ARM (Williston Lake)" → waterbody_key: "NATION ARM"
- "ELK RIVER'S TRIBUTARIES (see exceptions)" → waterbody_key: "ELK RIVER"
- "LITTLE CAMPBELL RIVER" → waterbody_key: "LITTLE CAMPBELL RIVER" (preserve "Little")
- ""GRASSY" LAKE (unnamed lake approx. 1 km southwest of West King Lake)" → waterbody_key: ""GRASSY" LAKE"
- "EAST (Fork) WHITE RIVER" → waterbody_key: "EAST WHITE RIVER" (note: "(Fork)" goes to alternate_names)

**IDENTITY_TYPE CLASSIFICATION**:

Classify the waterbody type based on the name. Valid types:
- STREAM: All flowing water bodies including rivers, creeks, streams, and their named parts
  * Examples: "ELK RIVER", "BURNT BRIDGE CREEK", "COQUIHALLA RIVER", "ATNARKO/BELLA COOLA RIVERS"
  * Note: "WATERSHED" in the name usually means ADMINISTRATIVE_AREA (a polygon boundary) - see ADMINISTRATIVE_AREA rules. Only use STREAM for "WATERSHED" if the regulation clearly applies to a single river and its tributaries being listed individually.
- STILL_WATER: All standing/non-flowing water bodies including lakes, reservoirs, ponds, sloughs, bays, arms
  * Examples: "KOOTENAY LAKE", "KINBASKET LAKE", "NATION ARM", "DAVIS BAY", "LEWIS SLOUGH", "DOWNTON LAKE (Reservoir)"
- ADMINISTRATIVE_AREA: Management areas, parks, watersheds, and jurisdictional boundaries (including "X WATERS" entries)
  * Examples: "CRESTON VALLEY WILDLIFE MANAGEMENT AREA", "STRATHCONA PARK WATERS", "KIKOMUN CREEK PARK", "LIARD RIVER WATERSHED", "CHILKOOT TRAIL NATIONAL HISTORIC PARK WATERS", "BOWRON LAKE Park waters other than Bowron Lake"
  * Keyword patterns: "PARK WATERS", "WILDLIFE MANAGEMENT AREA", "NATIONAL HISTORIC PARK", "WATERSHED (see map...)", "Park waters"
  * **CRITICAL**: ADMINISTRATIVE_AREA represents a GEOGRAPHIC POLYGON (park boundary, WMA boundary, watershed polygon) - NOT a specific waterbody
  * **includes_tributaries MUST be false**: Spatial containment replaces tributary logic - everything inside the polygon is covered
  * **DO NOT classify as ADMINISTRATIVE_AREA**: Dams (PROVOST DAM), specific ponds in parks (SARDIS PARK POND, JERRY SULINA PARK POND), specific lakes in parks (DEER LAKE in Sasquatch Park, LIGHTNING LAKE in Manning Park), specific creeks in parks (TROUT CREEK in Wells Gray Park). These are individual waterbodies WITHIN admin areas, not admin areas themselves.
  * **Rule of thumb**: If the title names a WATERBODY that happens to be in a park, use STREAM or STILL_WATER. If the title names the AREA/PARK/REGION itself, use ADMINISTRATIVE_AREA.
- CONFLUENCE: Specific confluence points where waterbodies meet (e.g., "SKEENA RIVER/KISPIOX RIVER CONFLUENCE")
- TRIBUTARIES: When title explicitly about tributary systems (e.g., "ELK RIVER'S TRIBUTARIES")
- MULTIPLE_WATERBODIES: When title covers multiple distinct NAMED waterbodies managed together (e.g., "WEAVER LAKE and WEAVER CREEK")
  * **CRITICAL**: Only use MULTIPLE_WATERBODIES when you can populate component_waterbodies with ACTUAL NAMES.
  * If waterbodies are "unnamed" (e.g., "UNNAMED LAKES"), you CANNOT populate component_waterbodies → use STILL_WATER instead.
  * Example: "UNNAMED LAKES (located immediately north and south of Bluey Lake)" → identity_type: STILL_WATER, not MULTIPLE_WATERBODIES.
    The parenthetical is a location_descriptor, not component names. The lakes have no individual names to list.

**COMPONENT_WATERBODIES EXTRACTION**:

For MULTIPLE_WATERBODIES type, extract the individual waterbody names into an array.

**WHAT TO EXTRACT**:
- Waterbodies separated by "/" or "and"
- Parse each component as a separate waterbody name
- Preserve the type suffix for each (RIVER, LAKE, CREEK, etc.)

**EXAMPLES**:
- "ATNARKO/BELLA COOLA RIVERS" → component_waterbodies: ["ATNARKO RIVER", "BELLA COOLA RIVER"]
- "WEAVER LAKE and WEAVER CREEK" → component_waterbodies: ["WEAVER LAKE", "WEAVER CREEK"]
- "SKEENA RIVER/KISPIOX RIVER" (when not CONFLUENCE) → component_waterbodies: ["SKEENA RIVER", "KISPIOX RIVER"]
- Single waterbody → component_waterbodies: []

**NOTE**: CONFLUENCE type is different - it's a specific geographic point where rivers meet, not a compound system. Use MULTIPLE_WATERBODIES when regulations apply to multiple waterbodies as a combined system.

**ALTERNATE_NAMES EXTRACTION**:

Extract ALL alternate/former/quoted names from parenthetical sections. This is an ARRAY of strings.

**WHAT TO EXTRACT**:
- Alternate official names: "(McNaughton)", "(Reservoir)", "(Irrigation Lake)"
- Former names: "(formerly Tseax River)", "(formerly known as Tatsamenie Lake's outlet streams)"
- Common names/nicknames: "(Sheep)", "(Bear)", "(Fork)", "(China)"
- Quoted names: "("Cameron")", "("Alces")", "("Vic Mawson")"
- "Also known as" variants: "(also known as Gravelpit Lake)"

**HOW TO FORMAT**:
- Each alternate name is a separate array element
- Remove the parentheses but keep the descriptive prefix if present
- Keep quotes in quoted names

**EXAMPLES**:
- "KINBASKET (McNaughton) LAKE" → alternate_names: ["McNaughton"]
- "KSI SII AKS RIVER (formerly Tseax River)" → alternate_names: ["formerly Tseax River"]
- "LEWIS ("Cameron") SLOUGH" → alternate_names: [""Cameron""]
- "NALTESBY LAKE (Bobtail Lake)" → alternate_names: ["Bobtail Lake"]
- "KWITZIL LAKE (also known as Gravelpit Lake)" → alternate_names: ["also known as Gravelpit Lake"]
- "EAST (Fork) WHITE RIVER" → alternate_names: ["Fork"]
- "SOWERBY ("Grundy") LAKE" → alternate_names: [""Grundy""]
- "GLACIER (Redslide) CREEK (unnamed tributary to Nanika River)" → alternate_names: ["Redslide"] (location goes elsewhere)
- Title with no alternates → alternate_names: []

**LOCATION_DESCRIPTOR EXTRACTION**:

Extract ONLY the disambiguating location information - phrases that help identify WHICH waterbody this is when multiple share the same name.

**WHAT TO EXTRACT** (into location_descriptor):
- Geographic proximity: "(near Kimberley)", "(Quadra Island)", "(North Vancouver)"
- Distance-based location: "(approx. 10 km west of 100 Mile House)", "(situated approx. 2.3 km northeast of Lorin Lake)"
- Watershed/drainage: "(Dog Creek drainage)", "(Canim Lake area)", "(Sustut Watershed)"
- Relative position: "(east of Allison Lake)", "(adjacent to West Road River)"
- Provincial parks: "(Crooked River Provincial Park)", "(Manning Park)"
- Geographic areas: "(Soda Creek area)", "(Fort St. James)", "(Sechelt)"
- Unnamed lake descriptions: "(unnamed lake located about 400 m west of Thalia Lake)"
- Part designations when NOT in core name: "(in Finlay Reach of Williston Lake)" - only if "DAVIS BAY" is the waterbody, not "Finlay Reach"

**WHAT NOT TO EXTRACT** (goes elsewhere):
- Scope information: "(upstream of X)", "(downstream of X)" → goes to global_scope
- Inclusions: "(includes X, Y)" → goes to inclusions
- Exceptions: "(see exceptions)", "(except X)" → goes to exclusions
- Alternate names: "(McNaughton)", "(formerly X)" → goes to alternate_names
- Notes/references: "(permit required...)", "(see map...)" → goes to notes
- Part names that ARE the core identifier: "MAIN BODY", "UPPER WEST ARM" → part of waterbody_key

**EXAMPLES**:
- "ECHOES LAKE (near Kimberley)" → location_descriptor: "near Kimberley"
- "BIG LAKE (approx. 10 km west of 100 Mile House)" → location_descriptor: "approx. 10 km west of 100 Mile House"
- "LARRY LAKE (unnamed lake located about 400 m west of Thalia Lake)" → location_descriptor: "unnamed lake located about 400 m west of Thalia Lake"
- "BOAR LAKE (Dog Creek drainage)" → location_descriptor: "Dog Creek drainage"
- "SQUARE LAKE (located in Crooked River Provincial Park)" → location_descriptor: "located in Crooked River Provincial Park"
- "ELK RIVER (upstream of Elko Dam)" → location_descriptor: null (scope, not location)
- "NATION ARM (Williston Lake)" → location_descriptor: "Williston Lake"
- No location info → location_descriptor: null

**NOTES EXTRACTION**:

Extract administrative/reference information that doesn't fit other categories.

**CRITICAL - PARENTHETICAL CONTENT CLASSIFICATION**:
Parenthetical content in the waterbody title should be classified as follows:
1. **Geographic/Spatial information** → global_scope, exclusions, or inclusions
   * Examples: "(upstream of X)", "(EXCEPT: Y)", "(includes Z)"
2. **Administrative/Reference information** → notes
   * Examples: "(see separate entries...)", "(permit required...)", "(see map on page...)"
3. **Location Disambiguators** → location_descriptor
   * Examples: "(near Kimberley)", "(approx. 10 km west of...)"
4. **Alternate Names** → alternate_names
   * Examples: "(McNaughton)", "(formerly Tseax River)"

**Rule of thumb**: If parenthetical content describes WHERE a rule applies or what's excluded/included, it's geographic (global_scope/exclusions/inclusions). If it provides administrative instructions, references, or meta-information about the regulations, it's notes.

**WHAT TO EXTRACT**:
- Map references: "(for location see map on page 34)", "(see map on page 42)"
- Permit notices: "(permit required see Note on page 34)"
- Cross-references: "(see also east White & North White Rivers)", "(see separate entries for these three waters)"
- Regulatory notices: "(Angling regulations... currently under review...)"
- Part descriptions when informational: "(all lakes in the park)"

**EXAMPLES**:
- "DUCK LAKE (permit required see Note on page 34)" → notes: "permit required see Note on page 34"
- "KOOTENAY LAKE - MAIN BODY (for location see map on page 34)" → notes: "for location see map on page 34"
- "WHITE RIVER (see also east White & North White Rivers)" → notes: "see also east White & North White Rivers"
- "ATNARKO/BELLA COOLA RIVERS EXCEPT: ... (see separate entries for these three waters)" → notes: "see separate entries for these three waters"
- "KIKOMUN CREEK PARK (all lakes in the park)" → notes: "all lakes in the park"
- No notes → notes: null

**EXCLUSIONS EXTRACTION**:

Extract geographic areas that are EXPLICITLY EXCLUDED from the waterbody definition. This is an ARRAY of ScopeObject items.

**THREE SOURCES FOR EXCLUSIONS**:

1. **FROM TITLE** - Direct exceptions in waterbody name:
   - "ATNARKO/BELLA COOLA RIVERS EXCEPT: Burnt Bridge Creek upstream of Sitkatapa Creek, Hunlen Creek upstream of Hunlen Falls"
   - "FRASER RIVER (except tributaries in parks)"
   - Parse each exception as a separate ScopeObject

2. **FROM REGULATION TEXT - "(see exceptions)" pattern**:
   - **CRITICAL**: Title phrase "(see exceptions)" means look for "EXCEPTIONS - SEE SEPARATE LISTINGS FOR:" in raw_regs text
   - Parse ALL waterbodies listed after this marker as exclusions
   - Each listed waterbody becomes a separate exclusion ScopeObject
   - Extract any scope modifiers (upstream/downstream/between) for each exception

3. **FROM REGULATION TEXT - "Does not include" pattern**:
   - Regulation text may state "Does not include [waterbody]" or similar phrasing
   - Extract the waterbody mentioned after "Does not include"
   - Parse as exclusion ScopeObject with appropriate scope type
   - Example: "Does not include the Kootenay River upstream from Kootenay Lake to the U.S. border near Creston"

**PARSING "SEE EXCEPTIONS" PATTERN**:

When waterbody name contains "(see exceptions)":
1. Search raw_regs for "EXCEPTIONS - SEE SEPARATE LISTINGS FOR:"
2. Extract the comma-separated list of waterbodies that follows
3. For each waterbody in the list:
   - Extract waterbody name
   - Extract any directional/scope modifiers ("upstream of", "downstream of", "between X and Y")
   - Create a ScopeObject with appropriate type (DIRECTIONAL, SEGMENT, WHOLE_SYSTEM, etc.)
4. Handle additional nested exceptions (e.g., "EXCEPT Coal Creek downstream of...")

**TRIBUTARY MARKING IN EXCEPTIONS - TRI-STATE LOGIC**:
Use the same tri-state logic as rule scopes:
- **true**: Exception text EXPLICITLY includes tributaries (e.g., "**[Includes Tributaries]**", "including tributaries")
  * Example: "Alexander Cr.**[Includes Tributaries]** upstream of..." → includes_tributaries: true
- **false**: Exception text EXPLICITLY excludes tributaries (e.g., "mainstem only", "excluding tributaries")
  * Example: "Burnt Bridge Creek mainstem only upstream of..." → includes_tributaries: false
- **null** (DEFAULT): Exception text is SILENT on tributary inclusion
  * Example: "Hunlen Creek upstream of Hunlen Falls" (no tributary mention) → includes_tributaries: null
  * Most exceptions will be null - only set true/false when explicitly stated

**NESTED EXCEPTIONS**:
- Text may have additional exceptions within the exceptions list (e.g., "EXCEPT Coal Creek downstream of old MF&M Railway Bridge")
- These are NOT exclusions - they exclude from the exclusion (double negative = inclusion)
- Handle carefully: if "ALL tributaries (EXCEPT X)" appears, X is actually INCLUDED in the main scope

**EXAMPLES - Title-based exclusions**:
- "ATNARKO/BELLA COOLA RIVERS EXCEPT: Burnt Bridge Creek upstream of Sitkatapa Creek" →
  exclusions: [{{type: "DIRECTIONAL", waterbody_key: "Burnt Bridge Creek", location_verbatim: "Burnt Bridge Creek upstream of Sitkatapa Creek", ...}}]
- "FRASER RIVER (except Harrison Lake)" →
  exclusions: [{{type: "WHOLE_SYSTEM", waterbody_key: "Harrison Lake", location_verbatim: "Harrison Lake", ...}}]
- No exclusions → exclusions: []

**INCLUSIONS EXTRACTION**:

Extract geographic areas that are EXPLICITLY INCLUDED in the waterbody definition. This is an ARRAY of ScopeObject items.

**WHEN TO USE**:
- Title contains "(includes X)", "(including X, Y, Z)"
- Title contains "(Includes the reservoirs behind...)"

Each inclusion becomes a ScopeObject with:
- type: Usually NAMED_PART or WHOLE_SYSTEM depending on the inclusion
- waterbody_key: The name of the included waterbody
- location_verbatim: The exact phrase from name_verbatim describing the inclusion
- includes_tributaries: Use TRI-STATE logic (true/false/null) based on whether inclusion text mentions tributaries
- Other fields as appropriate

**EXAMPLES**:
- "PEND D'OREILLE RIVER (Includes the reservoirs behind Waneta Dam and Seven Mile Dam)" →
  inclusions: [
    {{type: "NAMED_PART", waterbody_key: "PEND D'OREILLE RIVER", location_verbatim: "the reservoirs behind Waneta Dam and Seven Mile Dam", ...}}
  ]
- "SUMALLO RIVER (includes "Cedar" Lake, at Sunshine Valley)" →
  inclusions: [
    {{type: "WHOLE_SYSTEM", waterbody_key: ""Cedar" Lake", location_verbatim: ""Cedar" Lake, at Sunshine Valley", ...}}
  ]
- "VASEUX LAKE (including two lagoons on the west side of Okanagan River upstream of McIntyre Dam)" →
  inclusions: [
    {{type: "NAMED_PART", waterbody_key: "VASEUX LAKE", location_verbatim: "two lagoons on the west side of Okanagan River upstream of McIntyre Dam", ...}}
  ]
- No inclusions → inclusions: []

**VALIDATION**: Verify waterbody_key is derivable from name_verbatim by removing parenthetical sections and suffixes

TRIBUTARY INCLUSION LOGIC:

**CRITICAL - TWO DIFFERENT MODELS**:
The `includes_tributaries` field uses **different logic** depending on where it appears:

1. **GLOBAL SCOPE** (identity.global_scope) - **BOOLEAN (true/false)**:
   - This is the MASTER DEFAULT for the entire waterbody system
   - Always set to true or false (never null)
   - Inspect the input `symbols` array for markers like "Incl. Tribs", "Includes Tributaries"
   - Check waterbody names ending with "TRIBUTARIES" or "'S TRIBUTARIES"
   - **true**: If "Incl. Tribs" symbol present OR name ends with "TRIBUTARIES"
   - **false**: If no tributary markers found (explicit default)
   - **ADMINISTRATIVE_AREA EXCEPTION**: If identity_type is ADMINISTRATIVE_AREA, ALWAYS set includes_tributaries to false
     * Admin areas use spatial containment (polygon boundaries), not stream/tributary logic
     * Everything inside the polygon is covered regardless of tributary status

2. **ALL OTHER SCOPES** (exclusions, inclusions, rule scopes) - **TRI-STATE (true/false/null)**:
   - **true**: Text EXPLICITLY includes tributaries
     * Markers: "including tributaries", "**[Includes Tributaries]**", "on tributaries"
   - **false**: Text EXPLICITLY excludes tributaries
     * Markers: "mainstem only", "main stem only", "excluding tributaries"
   - **null** (DEFAULT): Text is SILENT on tributary inclusion
     * Most common - use when no tributary keywords appear
     * Application layer will resolve null → global value at query time

**GLOBAL SCOPE** (identity level):
- Inspect the input `symbols` array for markers like "Incl. Tribs", "Includes Tributaries", or similar
- If present, set global_scope.includes_tributaries to `true`
- Check waterbody names that END with or CONTAIN any of these patterns — ALL require `includes_tributaries: true`:
  * Ends with "TRIBUTARIES"
  * Ends with "'S TRIBUTARIES"
  * Ends with "'S TRIBUTARY STREAMS"
  * Ends with "TRIBUTARY STREAMS"
  * Contains "'S INLET & OUTLET STREAMS" (e.g., "WHITESWAN LAKE'S INLET & OUTLET STREAMS")
  * Contains "INLET & OUTLET STREAMS"
  * Contains "TRIBUTARY STREAMS"
- If NO tributary markers found, set to `false` (global default is explicit)
- **The global scope defines the DEFAULT tributary inclusion for all other scopes**

**EXCLUSIONS & INCLUSIONS** (identity level):
- Use tri-state logic based on the exception/inclusion text
- **true**: Exception text says "**[Includes Tributaries]**" or mentions tributary inclusion
- **false**: Exception text says "mainstem only" or mentions tributary exclusion  
- **null** (MOST COMMON): Exception text doesn't mention tributaries at all
- Example: "Burnt Bridge Creek upstream of X" (no mention) → includes_tributaries: null

**RULE SCOPE** (individual rule level) - TRI-STATE EXPLICIT OVERRIDE MODEL:

The `includes_tributaries` field at the rule scope level uses **tri-state logic** to eliminate implicit drift:

**THREE STATES**:
1. **true**: Rule text EXPLICITLY includes tributaries
   - Keywords: "including tributaries", "tributaries only", "on tributaries", "tribs"
   - Must find character-for-character evidence in rule_text_verbatim
   
2. **false**: Rule text EXPLICITLY excludes tributaries
   - Keywords: "mainstem only", "main stem only", "excluding tributaries", "on [River Name] only"
   - Must find character-for-character evidence in rule_text_verbatim
   
3. **null** (DEFAULT): Rule text is SILENT on tributary inclusion
   - No tributary-related keywords found in rule_text_verbatim
   - Rule inherits from global_scope.includes_tributaries at query time
   - **THIS IS THE MOST COMMON CASE** - use null unless text explicitly mentions tributaries

**PARSING LOGIC**:
1. Scan rule_text_verbatim for tributary keywords: "tributaries", "tribs", "mainstem", "main stem"
2. If keyword found + positive context ("including", "only on") → set to `true`
3. If keyword found + negative context ("excluding", "mainstem only") → set to `false`
4. If NO tributary keywords found → set to `null`
5. Validate: If true/false, the justifying substring MUST exist in rule_text_verbatim

**SPECIAL CASE - TRIBUTARIES_ONLY scope type**:
- When scope.type = "TRIBUTARIES_ONLY", the scope itself conveys tributary exclusivity
- In this case, includes_tributaries can be `null` (the type implies it)
- Or set to `true` if you want explicit confirmation

**CRITICAL ANTI-PATTERNS**:
- ❌ DON'T set based on restriction type (harvest vs vessel) - use null if text is silent
- ❌ DON'T "check back" with global scope when parsing - just look at rule text
- ❌ DON'T assume "Trout quota = 1" includes tributaries - use null if not mentioned
- ❌ DON'T assume "No powered boats" excludes tributaries - use null if not mentioned
- ✅ DO use null for most rules where tributary status isn't explicitly stated

RESTRICTION TYPES:

- closure: "No Fishing", fishing bans
- harvest: Catch limits, size limits, catch and release
- gear_restriction: Bait ban, fly fishing only, single barbless hook
- vessel_restriction: No powered boats, vessel speed limits
- licensing: Class II water, permits required
- note: Advisory information, references to other sections

ATOMIC REGULATION UNIT (ARU) ARCHITECTURE:

**CRITICAL**: Each rule contains EXACTLY ONE restriction with its dedicated scope (1:1 relationship).

**DISTRIBUTION PRINCIPLE**: When multiple restrictions apply to the same location, create MULTIPLE rules:
- Each rule has ONE restriction
- Multiple rules MAY share identical scopes (this is correct and intentional)
- Example: "Trout catch and release; bait ban; fly fishing only" all for "upstream of dam"
  * Create THREE separate rules (one for each restriction)
  * All three rules have the same scope ("upstream of dam")
  * This creates atomic traceability - if one restriction fails validation, you know exactly which one

**WHEN TO CREATE SEPARATE RULES**:
Create a SEPARATE rule for EACH unique restriction mentioned in the text, even if they share the same scope.

Example: "No fishing Apr 1-May 31; Bait ban; Single barbless hook"
- Rule 1: restriction="No Fishing", dates=["Apr 1-May 31"], scope=WHOLE_SYSTEM
- Rule 2: restriction="Bait ban", dates=null, scope=WHOLE_SYSTEM
- Rule 3: restriction="Single barbless hook", dates=null, scope=WHOLE_SYSTEM
- All three rules have identical scopes - this is correct

Example: "No fishing upstream. Bait ban downstream. Fly fishing only upstream."
- Rule 1: restriction="No Fishing", scope=UPSTREAM
- Rule 2: restriction="Bait ban", scope=DOWNSTREAM  
- Rule 3: restriction="Fly fishing only", scope=UPSTREAM
- Rules 1 and 3 share the same scope - this is correct

**SCOPE EQUALITY TEST** (for understanding, not for merging):
Two scopes are identical if ALL of these match:
1. type (both WHOLE_SYSTEM, both DIRECTIONAL, etc.)
2. location_verbatim (exact same text or both null)
3. landmark_verbatim (exact same text or both null)
4. direction (exact same direction or both null)
5. includes_tributaries (same value: both true, both false, or both null)

**NOTE**: Unlike the old architecture, identical scopes are ALLOWED and EXPECTED in ARU architecture.

**CRITICAL BUFFER vs DIRECTIONAL**:
A BUFFER scope (bounded distance) is NEVER equal to a DIRECTIONAL scope (unbounded), even if they share the same landmark and direction:
- "upstream of dam" (DIRECTIONAL, unbounded) ≠ "upstream 100 m from dam" (BUFFER, bounded)
- Keep these as separate rules

**CRITICAL TRIBUTARY LOGIC - TRI-STATE MODEL**:
The global_scope.includes_tributaries defines the DEFAULT, but rule scopes use **null to indicate silence**.

**Rule scope tri-state logic**:
- **DEFAULT**: Set includes_tributaries to `null` unless text explicitly mentions tributaries
  * `null` means "silent - inherits from global at query time"
  * Do NOT pre-compute inheritance - let the application layer handle it
  
- **EXPLICIT OVERRIDE - Only set true/false when**:
  * "TRIBUTARIES ONLY" → use TRIBUTARIES_ONLY scope type (can use includes_tributaries=true or null)
  * "on tributaries" / "including tributaries" → includes_tributaries=true (explicit in text)
  * "MAINSTEM ONLY" / "on mainstem" / "excluding tributaries" → includes_tributaries=false (explicit in text)
  * No tributary keywords found → **includes_tributaries=null** (the most common case)

**CRITICAL**: Setting null is NOT the same as inheriting - it's declaring "text is silent".
- Application layer will resolve null → global value at query time
- This prevents hallucination and makes overrides auditable
- Most rules will have includes_tributaries=null

Example: Header "Incl. Tribs" (global=true)
- "Trout quota = 1" → WHOLE_SYSTEM with includes_tributaries=null (silent)
- "No powered boats" → WHOLE_SYSTEM with includes_tributaries=null (silent, SAME scope - COMBINE!)
- "Bait ban on mainstem only" → WHOLE_SYSTEM with includes_tributaries=false (explicit "mainstem only" - SEPARATE!)

Example: Header NO "Incl. Tribs" (global=false)
- "Trout quota = 1" → WHOLE_SYSTEM with includes_tributaries=null (silent)
- "Catch and release on tributaries" → scope with includes_tributaries=true (explicit "on tributaries" - SEPARATE!)
- "Bait ban" → WHOLE_SYSTEM with includes_tributaries=null (silent, SAME as first rule - COMBINE!)
